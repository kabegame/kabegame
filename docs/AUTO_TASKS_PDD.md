# 自动运行配置与定时运行 — 产品需求文档（PDD）

## 0. 文档说明

| 项 | 内容 |
|----|------|
| **文档类型** | 产品需求 / 交互说明（实现见 `AUTO_TASKS_PLAN.md`） |
| **范围** | 在**现有运行配置**上增加「定时运行」能力；**不引入**独立的「定时任务」实体或新表概念。产品新增 **「自动运行配置」Tab**：配置**首先服务于自动运行**（定时规则与总开关为主叙事），并支持 **CRUD、复制**；用户也可**手动从该处加载并执行**（与定时共用同一套配置数据）。**开始收集对话框**不编辑配置，仅提供前往 **「自动运行配置」** 的入口，并在对话中**选择已保存的自动运行配置**触发一次收集 |
| **平台** | Android、Windows、macOS、Linux（与主应用一致；不涉及 iOS） |

---

## 1. 背景与目标

### 1.1 问题

用户希望**以自动运行为主**管理多套收集规则（何时跑、跑哪套插件参数），同时偶尔**手动**用同一套规则立刻跑一遍；需要能区分「手动加载执行」与「定时触发」的运行记录，并能从记录**回到该配置**调整或关闭定时。

### 1.2 目标（可验收）

- **「自动运行配置」** Tab 中管理的每条配置（底层仍为**一条运行配置**）最多拥有一套**定时运行**设置（**一个配置 = 至多一个定时规则**），字段挂在该配置记录上，不新增「定时任务」业务对象。
- 定时支持**三种模式**（见第 4 节）：循环周期、固定周期（每日）、**延迟运行**（多久之后执行**一次**）。
- 新建或首次开启定时时，**默认处于开启**状态（用户仍可关闭总开关）。
- 任务抽屉中，由**该配置的定时**触发的运行显示**闹钟图标**；**仅点击图标**跳转到**该运行配置**的编辑入口，并**聚焦定时运行**区块（或等价交互）。
- **持久化与重启**：**延迟一次**须保存**计划执行的绝对时刻**（真正到点运行时间），以便下次启动恢复倒计时或识别漏跑；**间隔**须在每次**定时触发**执行完成后记录**最后一次定时运行时间**，以便下次启动恢复「下一周期」节奏（详见 §4.1、§4.3）。
- **启动补跑**：若下次启动时发现存在「本应已执行却未执行」的定时项，**自动弹出确认**，用户可选择 **立即执行** 或 **暂不处理**（详见 §5.4）。

### 1.3 非目标（产品边界）

- 应用进程未存活或被系统挂起时，**不保证**在计划时刻**自动**执行；需在说明文案中让用户理解。  
- **补充**：**不**承诺系统级后台唤醒；但允许在**用户下次打开应用时**通过 §5.4 提示漏跑并询问是否补跑。

## 2. 核心概念（用户可见）

| 术语 | 说明 |
|------|------|
| **自动运行配置**（Tab / 用户心智） | 画廊（或收集相关区域）**新增 Tab**，集中管理「会按规则自动跑、也可手动点跑」的配置列表；**产品叙事上配置首先为自动运行服务**（定时区块与状态为列表/编辑的主信息），插件与参数是**同一条配置**的组成部分。 |
| **运行配置**（数据层） | 与「自动运行配置」**一一对应**的同一条持久化记录：插件、参数 + **定时运行**字段（`schedule_*`）。 |
| **定时运行** | 依附于该条配置的**设置区块**；同一配置**至多一个**定时规则（三模式互斥）。 |
| **运行记录 / 任务（抽屉）** | 每次执行一条记录；**来源**区分手动（含「从自动运行配置加载执行」）与定时触发，供闹钟与筛选；技术上关联 **`run_config_id`**。 |

## 3. 产品决策摘要

| 决策点 | 结论 |
|--------|------|
| 数据形态 | **不新增**独立定时任务实体；在**运行配置**上扩展字段 |
| 配置与定时数量关系 | **一个运行配置最多一个定时规则**（三种模式互斥，三选一） |
| 新建默认定时开关 | **默认开启**（在用户明确进入并保存定时设置的前提下；若「未配置任何定时参数」则视为未启用，见工程校验） |
| 任务抽屉闹钟 | **仅点击闹钟图标** → 打开该 **`run_config_id`** 对应配置的编辑，并定位到定时区块；标题/行主体仍为查看当次运行 |
| 第三种模式 | **延迟运行**：自保存或确认起计，**经过指定时长后执行一次**（非每周、非多任务） |
| **「自动运行配置」Tab** | **新增 Tab**，命名建议固定为 **「自动运行配置」**（i18n 键与文案另表）。承担配置的 **CRUD、复制** 与**定时编辑**；列表/详情强调**下次/周期、启用状态**等自动运行信息 |
| **手动执行** | 用户**主动**用某条配置跑收集：**①** Tab 内 **「立即运行」**（或「手动执行一次」）；**②** **开始收集对话框**中选择该配置后点「开始」。二者共用同一 `run_config_id`，运行记录标记为 **手动**，非定时 |
| **开始收集对话框**（`CrawlerDialog`） | **不**在此编辑配置；提供**前往「自动运行配置」Tab** 的入口。在此**仅选择**已有配置并**开始收集**（即手动从自动运行配置加载执行） |
| **延迟一次持久化** | 除用户输入的时长外，须持久化**计划执行时刻**（墙钟时间），用于重启恢复与漏跑判断 |
| **间隔持久化** | 每次**由定时触发**的运行完成后，回写**最后一次定时运行时间**；**手动「立即运行」不刷新该时间**，避免打乱间隔节奏 |
| **启动漏跑** | 启动时若检测到漏跑项，**弹窗**询问 **立即执行** / **暂不处理**；详见 §5.4；**立即执行**产生的运行记录为**手动**来源（非 scheduled） |

## 4. 调度模式（三种，互斥）

用户在**该运行配置**的「定时运行」区块内**必选其一**；切换模式时清空其他模式的专有字段（建议切换时轻量提示或确认，避免误清空）。

### 4.1 模式一：循环周期（间隔）

**定义**：从用户**保存定时设置且总开关为开**的时刻起算，按**固定时间间隔**重复触发。

**界面要素**

- 间隔数值 + 单位（分钟 / 小时 / 天；上限由工程与安全策略约束）。
- 人类可读摘要：如「每 6 小时运行一次」。

**行为说明**

- **首次触发**：建议与工程约定一致并在文案写清（例如「保存后先经过完整间隔再首次执行」或「保存后立即开始计间隔」二选一）。
- **重启恢复**：依赖持久化的 **「最后一次定时运行时间」**（仅统计 **Scheduled** 来源、在运行完成时回写，具体时点由工程在「开始执行」与「执行结束」间**择一统一**）。下次启动据此与间隔推算是否已跨过应触发时刻；若已跨过且应用未运行，进入 §5.4 漏跑流程。
- 与模式二、三互斥。

---

### 4.2 模式二：固定周期（每日，绝对时刻）

**定义**：在**每个自然日**，按用户选择的「时 / 分」规则在指定时刻触发（时区与设备一致）。

**粒度**

- **精确到分**。
- **分**：**某分** 或 **不指定分** → 不指定则视为 **0 分**。
- **时**：**每时**、**某时**、**不指定时** 三选一 → 不指定则视为 **0 时**。

**语义表（产品规则）**

| 时的选择 | 分的选择（不指定=0 分） | 每日触发含义（摘要文案建议） |
|----------|-------------------------|------------------------------|
| **每时** | 某分 *m* | 每天 **00:*m*、01:*m*、…、23:*m***（最多 24 次） |
| **每时** | 不指定分（0 分） | 每天 **00:00…23:00** 整点各一次 |
| **某时** *h* | 某分 *m* | 每天 **h:m** 一次 |
| **某时** *h* | 不指定分（0 分） | 每天 **h:00** 一次 |
| **不指定时**（0 时） | 某分 *m* | 每天 **00:*m*** 一次 |
| **不指定时**（0 时） | 不指定分（0 分） | 每天 **00:00** 一次 |

**界面建议**

- **时**：「每时 / 指定时刻 / 不指定（0 点）」；指定时刻时 0–23。
- **分**：0–59 + 「不指定（0 分）」等价交互。
- **实时预览**一行，避免「每时 / 某时」混淆。

---

### 4.3 模式三：延迟运行（一次性）

**定义**：从用户**保存定时设置且总开关为开**的时刻起，经过用户指定的**一段时长**后，**自动执行一次**收集；**执行完成后，该次延迟调度即结束**（不再按同一延迟重复）。

**界面要素**

- 延迟数值 + 单位（建议：分钟 / 小时；是否支持「天」由工程决定）。
- 摘要示例：「保存后 30 分钟运行一次」。

**行为说明（建议，需与工程一致）**

- **持久化计划时刻**：保存（或重算）延迟规则时，须写入**真正的计划执行时刻**（墙钟时间戳），而非仅在内存中倒计时；下次启动用该时刻恢复剩余等待或判定漏跑（与 `AUTO_TASKS_PLAN.md` §2.1 一致）。
- **到点执行一次**后：自动**清除「延迟运行」模式**或**关闭定时总开关**（二选一，推荐：**执行成功后关闭定时或切回「未选择模式」空态**，避免用户误以为仍会再延迟一次）；并清除已持久化的计划执行时刻。
- 若在延迟期间用户**再次保存**定时设置：以**最新保存时间**重新起算延迟（覆盖上一轮未触发的等待），并**更新**计划执行时刻。
- 若在延迟期间用户**关闭定时总开关**：取消待触发的一次执行；清除或忽略计划执行时刻。
- 与模式一、二互斥。

## 5. 用户使用链路

### 5.1 开始收集对话框（`CrawlerDialog`）

1. **不**在对话框内编辑配置；用户在此从**已保存的自动运行配置**列表中**选择一条**并**开始收集**，即 **手动从自动运行配置加载执行**（`trigger_source = manual`，与 Tab 内「立即运行」等价数据源）。
2. 提供入口：**「管理自动运行配置」** / **「前往自动运行配置」** → 切换到 **「自动运行配置」Tab**（或等价路由）。
3. 若无可用配置：短文案引导用户先到 Tab 内**新建**，再回对话框选择。

### 5.2 「自动运行配置」Tab（主阵地：自动运行 + CRUD + 手动执行）

1. **定位**：配置 **首先服务于自动运行** — 列表优先展示 **定时摘要、启用状态、上次自动运行结果** 等；插件/参数在编辑流中作为「这条自动规则跑什么」的组成。
2. **列表**：已保存配置；支持 **创建、编辑、删除**；**复制（copy）** — 新 id，内容派生自源配置（定时字段是否一并复制：**择一写死**并在复制说明中告知用户）。
3. **手动执行**：列表行或详情提供 **「立即运行」**（或「手动执行一次」）→ 用当前配置 **立刻** 启动一次收集，**不计入**定时触发（运行记录为手动来源）。
4. **编辑页 / 抽屉**：插件与参数 + **「定时运行」** 区块（三模式、总开关）；保存写入同一条记录；**总开关默认开**等与 §3、§4 一致。
5. 保存成功：若定时已启用，可提示「应用需保持运行才会在定时到达时执行」。
6. **可选**：筛选「仅已启用定时」；数据源仍为单表运行配置，**不**新建实体。

### 5.3 从任务抽屉回溯

1. 由**定时**触发的运行：标题前 **闹钟图标**（无障碍：「由定时运行触发，配置：xxx」）。**手动**从自动运行配置执行（Tab「立即运行」或开始收集对话框选配置）**不**显示闹钟。
2. **仅点击闹钟**：打开 **对应配置** 在 **「自动运行配置」Tab** 内的编辑视图，并**滚动/聚焦到「定时运行」**（与 §5.2 为同一编辑入口）。
3. 点击行其余区域：仍为查看当次运行详情。
4. 若配置已删除：提示「关联的配置已不存在」，可提供留在任务列表或进入配置列表空态引导。

### 5.4 应用启动与漏跑补执行

1. **时机**：用户**打开应用并完成配置数据加载后**（与工程挂载点一致），扫描所有**已启用定时**的运行配置，判断在**进程未运行期间**是否有**应按定时（Scheduled）执行却未执行**的项。  
2. **漏跑判定（产品语义）**  
   - **延迟一次**：当前时间 **已达到或超过** 持久化的计划执行时刻，且该次延迟尚未成功执行、规则未被用户取消。  
   - **间隔**：按「最后一次定时运行时间 + 间隔」判断，在离线期间**至少错过一轮**应触发（与 §4.1 工程语义一致）。  
   - **每日**：在离线期间错过的计划槽位；若多次错过，可与工程约定**合并展示**（例如仅列「最近一次本应执行时间」或摘要「已错过 N 次」），避免弹窗过长。  
3. **弹窗**：若存在一条及以上漏跑项，**自动弹出**模态确认（非静默）：说明概况，主按钮 **「立即执行」**，次按钮 **「暂不处理」**。  
4. **立即执行**：对列出的漏跑项**启动收集**，执行链路与「从配置起跑」一致，但运行记录视为 **手动**（与 Tab「立即运行」相同来源语义）：**不**记为定时触发，任务抽屉**不**显示闹钟。执行完成后更新与定时规则相关的**持久化状态**（如清除已触发的延迟一次、避免同一漏跑重复提示等，见工程计划），**不**将本次补跑记为「由定时触发」的历史。  
5. **暂不处理**：**不**自动启动收集；恢复**后续**定时调度（下一周期或下一槽位由工程按 §4 计算），**不**将已错过的执行记为已完成。  
6. **无漏跑**：不弹窗。同一启动会话内，处理完毕后**不应**对同一漏跑重复弹窗。

## 6. UI / 交互细则

### 6.1 总开关与模式

- **一个配置仅一套定时字段**；切换模式会替换规则类型，列表上不会出现「同一配置两行定时」。
- 删除运行配置：定时字段随配置一并删除，无需单独删「任务」。

### 6.2 与插件推荐（若启用）

- 插件可推荐默认间隔等，在 **「自动运行配置」Tab** 编辑某条配置时**预填**「定时运行」表单；持久化的是用户确认后的值。产品默认「定时总开关打开」仍以 §3 为准；若插件建议关闭，在 UI 上可显示说明，由用户决定。

### 6.3 下次运行倒计时进度条

在 **「自动运行配置」Tab 列表** 中，每条**已启用定时**的配置显示一个**倒计时进度条**，直观反映「距离下次自动运行还有多久」。

**进度条总量（满格 = 一个完整周期）**

| 模式 | 进度条总量 | 说明 |
|------|-----------|------|
| **循环周期** | `schedule_interval_secs`（用户设定的间隔） | 如间隔 6 小时 → 总量 6h |
| **延迟运行** | `schedule_delay_secs`（用户设定的延迟时长） | 从保存时刻到计划执行时刻的完整时长 |
| **每日（某时 h:m）** | **24 小时**（86400 秒） | 每天执行一次，周期为一天 |
| **每日（每时 + 某分 m）** | **1 小时**（3600 秒） | 每小时执行一次，周期为一小时 |

**已消耗量（进度条当前填充）**

- **循环周期**：`now - schedule_last_run_at`（若无上次运行则 `now - 配置保存时刻`）。
- **延迟运行**：`now - (schedule_planned_at - schedule_delay_secs)`，即从保存时刻起已过去的时间。
- **每日**：`now - 上一个触发槽位时刻`（若当天尚未触发过，则为 `now - 前一天最后一个槽位`）。

进度百分比 = `min(已消耗量 / 总量, 1)`。

**前端更新策略**

- 纯前端 `setInterval`（建议 1 秒）驱动，基于配置数据中的时间戳实时计算；不依赖后端推送。
- 进度条旁可附带**剩余时间文字**（如「还剩 2 小时 15 分」）。

**配置变更时的行为**

- 用户**保存**配置（修改了模式、间隔、时刻等定时参数）后：
  - 进度条**归零**（已消耗量重置为 0）。
  - 总量**同步更新**为新参数对应的值。
  - 原因：保存后后端 Scheduler 以新参数重新计算 `next_fire_at`，旧进度已无意义。
- 用户**关闭定时总开关**后：进度条**隐藏**。

**不显示进度条的情况**

- `schedule_enabled = false`（定时未开启）。
- `schedule_mode` 为 `NULL`（未设置模式）。
- `delay_once` 已执行完毕（`schedule_enabled` 已被关闭）。

### 6.4 空状态与说明

- 未开启定时：区块可折叠，主文案「未设置定时运行」。
- 「应用需保持运行」：首次开启定时或帮助入口简短展示即可；可与 §5.4 配合说明「未运行时不会自动执行，**下次打开应用**可提示是否补跑」。

### 6.5 启动漏跑弹窗

- 文案需说明：**因应用当时未运行**，部分自动运行未在计划时刻执行；避免让用户误以为被静默跳过。  
- 按钮含义清晰：**立即执行**（补跑）、**暂不处理**（仅恢复后续节奏）。  
- **立即执行**补跑的任务在抽屉中与**手动运行**一致，**无**闹钟图标。  
- Android：**模态返回键**行为与项目弹窗规范一致。

## 7. 文案与无障碍（要点）

- 「每时」与「某时」须有简短释义或帮助链。
- 「延迟运行」须标明 **只执行一次**，避免与「间隔循环」混淆。
- 闹钟图标：`aria-label` 含配置名称或「在自动运行配置中打开定时设置」。
- 漏跑弹窗：主/次按钮有无障碍名称，列表项可读（配置名 + 简要时间说明）。
- 倒计时进度条：无障碍文本应包含「距下次运行还剩 X 分钟」等可读描述。

## 8. 验收标准（产品侧）

- [ ] 定时字段挂在**运行配置**上，**无**独立定时任务实体（产品语义）。
- [ ] 同一配置**仅**能保存一种定时模式；切换模式后旧模式参数不再生效。
- [ ] 三种模式行为符合第 4 节（含延迟**仅一次**及执行后的状态处理）。
- [ ] 新建并保存定时后，**默认总开关为开**（在参数合法前提下）。
- [ ] 产品含 **「自动运行配置」Tab**；配置叙事以**自动运行**为主，并具备 **CRUD**、**复制**、**立即运行（手动）**。
- [ ] **开始收集对话框**不编辑配置；提供前往该 Tab 的入口；对话框内可选配置并**开始**＝手动加载执行。
- [ ] 任务抽屉仅**点击闹钟**进入 **Tab 内**该条配置的编辑并定位定时区块；其他点击行为不变。
- [ ] **仅定时触发**的运行显示闹钟；**手动**执行（Tab 立即运行 / 对话框开始）不显示闹钟。
- [ ] 固定周期（每日）组合与第 4.2 节表一致。
- [ ] **延迟一次**持久化**计划执行绝对时刻**；重启后可恢复剩余等待或正确识别漏跑。
- [ ] **间隔**在每次定时触发执行完成后更新**最后一次定时运行时间**；重启后可恢复下一周期；手动执行不篡改该时间。
- [ ] **「自动运行配置」列表**中已启用定时的配置显示**倒计时进度条**，总量与模式对应（§6.3）；前端定时刷新。
- [ ] 配置变更（保存新定时参数）后进度条**归零**并更新总量；关闭定时后进度条**隐藏**。
- [ ] 启动时若存在漏跑项，出现 §5.4 / §6.5 所述弹窗；**立即执行**与**暂不处理**行为符合该节描述。
