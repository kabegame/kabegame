## 一、项目概述

Kabegame 是一个跨平台的视觉素材收集与壁纸管理应用，主要面向需要长期收集、整理、预览和复用图片或视频素材的用户。项目以本地媒体库为核心，结合插件化内容获取、画册管理、壁纸设置与轮播、虚拟磁盘访问、任务管理和命令行工具等功能，尝试将分散的素材收集和桌面壁纸管理流程整合到一个统一的软件中。

在实际使用场景中，用户通常需要同时使用插件、下载工具、文件管理器、图片查看器和系统壁纸设置工具。随着素材数量增加，文件夹会变得难以维护，重复图片、来源信息、分类关系和壁纸使用历史也不容易管理。Kabegame 的开发目标是提供一个本地优先、可扩展、可维护的媒体管理工具，使用户能够更高效地完成“获取素材、整理素材、预览素材、应用素材”的完整流程。

本项目具有较强的工程实践特点：它不仅包含桌面端图形界面，还涉及 Rust 后端核心、Tauri 跨平台封装、Vue 前端界面、Rhai 插件脚本、CLI 命令行工具、Android 预览版、跨平台打包和插件生态维护等内容。

## 二、项目背景与问题分析

视觉素材管理是一个常见但容易被低估的需求。对于壁纸收藏者、设计参考收集者、二次元图片爱好者以及需要整理大量图片素材的用户来说，素材的来源、分类、预览和复用都需要长期维护。传统方式通常依赖普通文件夹和手动命名，短期内可行，但当素材规模扩大后会出现以下问题：

1. 素材获取方式分散，不同网站、不同文件格式和不同下载逻辑需要分别处理。
2. 本地文件夹缺少结构化管理能力，难以记录来源、标签、画册关系和使用历史。
3. 图片预览、批量导入、去重整理和壁纸设置分别依赖不同工具，操作流程较长。
4. Windows、macOS、Linux 和 Android 等平台之间存在差异，用户很难获得一致体验。
5. 独立脚本虽然灵活，但缺少统一的配置、日志、任务管理和安全边界。

因此，本项目的机会在于：在简单壁纸软件和专业数字资产管理系统之间提供一个更聚焦的本地工具。Kabegame 不追求一开始就建设大型云端服务，而是优先做好本地媒体库、插件获取、自动化和跨平台体验。

## 三、项目目标

本项目的总体目标是将 Kabegame 建设为一个专业级、本地优先、插件可扩展的视觉素材管理工具。

主要目标如下：

1. 提供统一的素材收集、导入、整理、预览和壁纸应用流程。
2. 通过画册、收藏、元数据、任务历史和自动化功能减少用户的手动文件管理成本。
3. 支持 Windows、macOS、Linux 和 Android 等平台，尽量保持核心体验一致。
4. 建立插件化内容获取机制，使新增素材来源或导入流程不需要频繁修改主程序。
5. 同时提供 GUI 和 CLI，满足普通用户、批处理和自动化使用需求。
6. 保持开源项目属性，并为未来的插件分发、技术支持、赞助或企业打包等可持续方向预留空间。

## 五、系统总体设计

Kabegame 采用前后端分离但统一打包的桌面应用架构。前端负责用户界面和交互，后端负责文件系统、下载任务、插件运行、壁纸设置、虚拟磁盘和本地数据处理。桌面端通过 Tauri 将前端页面和 Rust 后端能力整合成跨平台应用。

系统可分为以下几层：

1. 用户界面层：基于 Vue 3、TypeScript、Element Plus、Vant 等技术实现画廊、画册、插件管理、任务管理和设置页面。
2. 应用服务层：通过 Tauri IPC 调用后端能力，负责连接界面操作与核心业务逻辑。
3. 核心业务层：由 Rust 实现，处理媒体库、任务、插件、导入、壁纸和虚拟磁盘等核心逻辑。
4. 插件扩展层：通过 `.kgpg` 插件包和 Rhai 脚本提供可扩展的素材获取能力。
5. 平台适配层：处理 Windows、macOS、Linux 和 Android 上不同的壁纸、文件系统、权限和打包行为。

主要模块设计如下：

| 模块 | 主要职责 |
|---|---|
| 画廊模块 | 展示本地媒体库中的图片和视频，支持分页、预览、多选和基础管理。 |
| 画册模块 | 提供素材分类、排序、封面设置和壁纸轮播来源管理。 |
| 插件模块 | 管理插件安装、配置、运行、更新和删除，并通过脚本扩展素材来源。 |
| 任务模块 | 统一管理爬取、导入和下载任务，展示进度、日志和结果。 |
| 壁纸模块 | 负责图片 / 视频壁纸设置和按画册轮播。 |
| 虚拟磁盘模块 | 将画册结构映射为虚拟盘或虚拟文件夹，方便用户通过文件系统访问素材。 |
| CLI 模块 | 提供命令行入口，支持批处理、插件执行和自动化脚本集成。 |
| 文档与国际化模块 | 提供多语言说明、插件开发指南和用户帮助内容。 |

## 六、关键技术与实现思路

本项目主要采用 Tauri 作为跨平台桌面应用框架。Tauri 的优势在于可以使用 Web 技术构建界面，同时利用 Rust 提供较高性能和系统能力访问。前端使用 Vue 3 和 TypeScript 实现组件化界面，配合 Pinia、Vue Router、Element Plus 等工具完成状态管理、页面路由和交互组件。

后端核心采用 Rust 实现，适合处理本地文件、并发任务、下载、平台能力调用和命令行工具。插件系统使用 Rhai 脚本语言，使插件作者能够以较低成本编写爬取逻辑。插件包以 `.kgpg` 形式分发，包含元数据、配置文件、脚本和文档。这样可以降低主程序对具体内容来源的依赖，当某个网站结构变化时，可以优先更新对应插件，而不是修改整个应用。

项目还区分 Standard 和 Light 两类桌面构建。Standard 版本包含虚拟磁盘和 CLI 等完整能力，适合深度使用；Light 版本去掉部分系统依赖，安装更轻量，适合只需要基础功能的用户。这种分层发布方式可以降低不同用户的安装门槛。

## 七、开发过程

### 7.1 前期分析阶段

在前期阶段，项目首先明确了目标用户和核心问题。项目定位不是单纯的壁纸切换器，而是围绕视觉素材管理建立完整工作流。该阶段主要完成了项目背景分析、用户痛点整理、功能范围定义和可行性判断。

通过分析可知，项目最重要的能力不是单个界面功能，而是“本地媒体库 + 插件获取 + 自动化使用”的组合。因此，开发计划需要优先保证核心媒体管理流程稳定，再逐步扩展插件生态和平台能力。

### 7.2 需求设计阶段

需求设计阶段将项目拆分为媒体库、画廊、画册、插件、任务、壁纸、虚拟磁盘、CLI、文档和发布等模块。每个模块既有独立职责，也需要与其他模块协同。例如，插件任务下载的图片需要进入媒体库，媒体库中的图片可以加入画册，画册又可以作为壁纸轮播和虚拟磁盘目录结构的来源。

该阶段还明确了非功能需求，包括跨平台兼容性、插件安全、数据可靠性、大图库性能、版本发布和文档质量等。这些内容决定了项目不能只关注功能实现，还必须关注长期维护和用户信任。

### 7.3 核心实现阶段

核心实现阶段主要围绕以下工作展开：

1. 建立桌面应用基础结构，实现前端界面与 Rust 后端之间的调用。
2. 实现画廊浏览和本地媒体管理，使用户能够导入、查看和整理素材。
3. 实现画册功能，为素材分类、壁纸轮播和虚拟目录提供基础。
4. 建设插件运行机制，使不同来源的素材获取可以通过插件扩展。
5. 建设任务管理功能，对爬取和导入任务提供进度、日志和结果查看。
6. 实现壁纸设置、自动轮播、虚拟磁盘和 CLI 等增强功能。
7. 针对不同系统准备安装包、轻量版和标准版发布方案。

在实现过程中，项目需要处理较多跨平台差异。例如 Windows 虚拟磁盘依赖 Dokan，macOS 虚拟磁盘依赖 macFUSE，Linux 需要处理 FUSE 和桌面环境差异，Android 则需要考虑移动端权限和界面适配。

### 7.4 文档与发布阶段

为了降低用户使用和开发者参与成本，项目需要提供安装说明、功能说明、插件开发指南、Rhai API 文档和常见问题排查说明。文档不仅服务最终用户，也服务插件作者和潜在贡献者。

发布阶段需要分别考虑 Windows、macOS、Linux、Android、CLI、Standard 和 Light 构建。为了提高发布质量，应建立发布检查清单，包括构建成功、基础功能可用、插件安装和运行正常、壁纸功能可用、文档更新完成等。

## 九、风险分析与应对措施

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 跨平台行为不一致 | 可能导致某些系统上壁纸、虚拟磁盘或安装包不可用 | 建立平台测试矩阵，分平台维护适配逻辑 |
| 插件安全风险 | 插件可能访问网络或本地数据，影响用户信任 | 增加插件校验、权限提示、禁用和删除机制 |
| 版权与合规风险 | 用户下载的素材可能涉及版权或网站条款问题 | 明确用户责任，发布内容使用说明，避免内置违规来源 |
| 维护成本过高 | 多平台、多模块和插件生态会增加开发压力 | 控制功能范围，优先稳定核心流程，鼓励社区贡献 |
| 大图库性能下降 | 素材数量增加后可能出现加载慢、内存高等问题 | 使用分页、缓存、索引和性能测试优化 |
| 数据迁移失败 | 版本升级可能损坏媒体库或元数据 | 增加迁移测试、备份建议和回滚方案 |
| 产品定位不清 | 用户可能只把项目理解为普通壁纸软件 | 在文档和主页中突出本地媒体管理与插件扩展能力 |
| 商业可持续性不确定 | 开源项目可能缺少长期维护资金 | 先提高可靠性和用户规模，再探索赞助、支持或插件服务 |

## 十一、存在问题与改进方向

虽然项目已经具备较完整的基础能力，但仍存在一些需要后续改进的问题：

1. 插件安全边界还需要进一步明确，例如插件权限、来源可信度和异常行为处理。
2. 跨平台测试成本较高，需要建立更稳定的自动化检查和人工验收流程。
3. 大规模媒体库下的性能仍需要通过真实数据持续验证。
4. 文档需要持续更新，尤其是安装问题、插件开发示例和常见错误处理。
5. 项目范围需要控制，短期内应优先稳定本地媒体库、插件系统和发布质量，不宜过早扩展复杂云端功能。

后续可以从以下方向继续推进：

1. 完善插件商店或插件索引，增加版本兼容信息和安全提示。
2. 优化缩略图缓存、分页浏览和元数据查询，提升大图库体验。
3. 建立发布检查清单，提升 Standard、Light、CLI 和 Android 包的发布稳定性。
4. 增强用户帮助系统，降低新用户的安装和使用门槛。
5. 在产品稳定后探索赞助、付费支持或托管插件分发等可持续发展方式。
