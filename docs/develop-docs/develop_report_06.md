## 一、文档目的

本文档用于说明 Kabegame 项目的核心需求，重点围绕三个主要功能模块展开：

1. 插件扩展
2. 壁纸轮播
3. 画廊管理

需求文档的作用是明确系统需要实现什么功能、面向哪些用户、需要满足哪些质量要求，以及后续开发和测试应如何判断功能是否完成。本文档不详细描述具体代码实现，而是从用户需求和系统功能角度给出规格说明。

## 二、项目概述

Kabegame 是一个跨平台视觉素材收集与壁纸管理应用。项目面向需要长期收集、整理、预览和使用图片或视频素材的用户，尤其适合壁纸收藏用户、二次元图片用户、插件开发者和需要批量管理素材的高级用户。

Kabegame 的核心定位是：通过插件扩展获取素材，通过画廊管理组织素材，通过壁纸轮播使用素材，从而形成“收集 - 整理 - 预览 - 应用”的完整工作流。

## 三、用户角色

| 用户角色 | 说明 | 主要需求 |
|---|---|---|
| 普通用户 | 希望简单导入图片、浏览图片并设置壁纸 | 操作简单、界面清晰、错误提示明确 |
| 壁纸收藏用户 | 拥有大量图片或视频素材 | 画廊管理、画册分类、壁纸轮播 |
| 插件使用者 | 通过插件从不同来源获取素材 | 插件安装、配置、运行、更新 |
| 插件开发者 | 为新网站或新导入流程编写插件 | 插件格式清晰、接口稳定、文档完整 |
| 高级用户 | 需要批量处理或自动化 | 任务日志、批量操作、CLI 扩展 |

## 四、业务目标

Kabegame 的业务目标包括：

1. 提供统一的视觉素材管理入口，减少用户在浏览器、文件夹、图片查看器和系统设置之间反复切换。
2. 支持插件化素材获取，使新素材来源可以独立扩展，不依赖主程序频繁更新。
3. 提供稳定的画廊和画册管理能力，使大量图片和视频能够被有序管理。
4. 支持图片或视频壁纸设置与自动轮播，使素材库能够直接转化为桌面展示效果。
5. 保持本地优先，用户的媒体文件和索引数据主要存储在本地。

## 五、总体功能范围

本项目核心功能范围如下：

| 模块 | 范围说明 |
|---|---|
| 插件扩展 | 插件安装、配置、运行、更新、删除、任务结果查看 |
| 画廊管理 | 本地导入、分页浏览、预览、筛选、排序、删除、画册关联 |
| 壁纸轮播 | 设置壁纸、选择轮播来源、随机/顺序轮播、轮播间隔配置 |
| 任务管理 | 展示插件或导入任务的进度、日志、结果和失败原因 |
| 元数据索引 | 记录媒体文件、插件来源、画册关系、壁纸历史等信息 |

本文档重点说明前三个模块，任务管理和元数据索引作为支撑能力一并描述。

## 六、功能需求一：插件扩展

### 6.1 需求说明

插件扩展模块用于解决素材来源多样化的问题。用户不应只能依赖主程序内置来源，而应能够通过插件扩展新的图片网站、本地导入方式或其他素材获取流程。

插件应以独立包形式分发，用户可以通过插件商店、插件索引或本地 `.kgpg` 文件安装。插件运行后产生的图片或视频应进入统一媒体库，并可继续被画廊、画册和壁纸轮播使用。

### 6.2 功能需求

| 编号 | 需求名称 | 需求描述 | 优先级 |
|---|---|---|---|
| PE-01 | 插件列表 | 系统应展示已安装插件列表，包括名称、版本、描述和状态 | 高 |
| PE-02 | 插件安装 | 系统应支持从插件商店或本地 `.kgpg` 文件安装插件 | 高 |
| PE-03 | 插件删除 | 用户应能够删除不需要的插件 | 高 |
| PE-04 | 插件启用 / 禁用 | 用户应能够临时禁用插件，禁用后插件不可运行 | 中 |
| PE-05 | 插件配置 | 插件应能声明配置项，例如页数、关键词、质量、分类等 | 高 |
| PE-06 | 插件运行 | 用户应能选择插件并启动素材收集任务 | 高 |
| PE-07 | 任务进度 | 插件运行时应显示进度、状态、开始时间和结束时间 | 高 |
| PE-08 | 错误提示 | 插件失败时应显示可理解的错误信息和任务日志 | 高 |
| PE-09 | 插件更新 | 系统应支持检查插件新版本并更新 | 中 |
| PE-10 | 插件文档 | 插件应提供使用说明、参数说明和网络要求 | 中 |
| PE-11 | 插件来源记录 | 通过插件获取的素材应记录插件 ID、来源 URL 和任务 ID | 高 |
| PE-12 | 插件安全提示 | 插件访问网络或本地文件时，应提供必要的权限提示 | 中 |

### 6.3 插件运行流程

1. 用户进入插件页面。
2. 用户安装或选择已有插件。
3. 系统展示插件说明和配置项。
4. 用户填写配置并点击开始收集。
5. 系统创建任务并显示运行进度。
6. 插件下载或导入素材。
7. 素材进入媒体库并显示在画廊中。
8. 用户可将结果加入画册或用于壁纸轮播。

### 6.4 异常需求

| 场景 | 系统行为 |
|---|---|
| 插件文件损坏 | 应提示插件无效，安装失败 |
| 插件版本不兼容 | 应提示版本不兼容，不应强制运行 |
| 网络连接失败 | 应显示网络错误，并保留任务日志 |
| 目标网站返回 403 | 应提示访问受限或插件可能失效 |
| 用户取消任务 | 应停止后续下载，并保留已完成结果 |
| 插件运行异常 | 应捕获异常，不应导致主程序崩溃 |

## 七、功能需求二：画廊管理

### 7.1 需求说明

画廊管理模块是 Kabegame 的核心媒体浏览入口。所有本地导入或插件获取的图片和视频都应进入画廊。用户可以在画廊中查看、预览、筛选、排序、删除和组织素材。

画廊不能只是简单文件夹列表，而应基于媒体库和元数据索引提供结构化管理能力。

### 7.2 功能需求

| 编号 | 需求名称 | 需求描述 | 优先级 |
|---|---|---|---|
| GM-01 | 本地导入 | 系统应支持导入本地图片、视频、文件夹和压缩包 | 高 |
| GM-02 | 媒体展示 | 系统应以网格形式展示媒体缩略图 | 高 |
| GM-03 | 分页浏览 | 画廊应支持分页或等效的分批加载机制 | 高 |
| GM-04 | 图片预览 | 用户应能打开图片进行大图预览、缩放和切换 | 高 |
| GM-05 | 视频识别 | 系统应能识别视频素材并区别于图片显示 | 中 |
| GM-06 | 多选操作 | 用户应能批量选择素材并执行操作 | 高 |
| GM-07 | 删除素材 | 用户应能删除媒体库中的素材，并获得确认提示 | 高 |
| GM-08 | 加入画册 | 用户应能将选中素材加入一个或多个画册 | 高 |
| GM-09 | 筛选 | 系统应支持按插件来源、日期、媒体类型、画册、尺寸等条件筛选 | 高 |
| GM-10 | 排序 | 系统应支持按导入时间、文件名、尺寸、壁纸设置时间等排序 | 中 |
| GM-11 | 元数据显示 | 用户应能查看文件名、路径、尺寸、格式、来源等基础信息 | 中 |
| GM-12 | 重复处理 | 系统应提供重复素材识别或提示能力 | 中 |
| GM-13 | 大图库性能 | 大量素材下画廊仍应保持可接受的加载速度 | 高 |

### 7.3 画册管理需求

画册是画廊管理的重要组成部分，用于将素材组织成可复用集合。

| 编号 | 需求名称 | 需求描述 | 优先级 |
|---|---|---|---|
| AL-01 | 创建画册 | 用户应能创建新画册并设置名称 | 高 |
| AL-02 | 编辑画册 | 用户应能修改画册名称、描述和封面 | 中 |
| AL-03 | 添加素材 | 用户应能从画廊或任务结果中将素材加入画册 | 高 |
| AL-04 | 移除素材 | 用户应能从画册中移除素材，但不一定删除原文件 | 高 |
| AL-05 | 画册排序 | 用户应能调整画册内素材顺序 | 中 |
| AL-06 | 轮播来源 | 画册应能作为壁纸轮播的来源 | 高 |

### 7.4 画廊管理流程

1. 用户导入本地素材或运行插件任务。
2. 系统将素材写入媒体库并生成缩略图。
3. 用户在画廊中分页浏览素材。
4. 用户通过筛选和排序缩小范围。
5. 用户预览图片或视频。
6. 用户将喜欢的素材加入画册。
7. 用户可以删除无用素材或将画册设置为壁纸轮播来源。

## 八、功能需求三：壁纸轮播

### 8.1 需求说明

壁纸轮播模块用于将媒体库中的素材应用到桌面或移动端壁纸。它是 Kabegame 区别于普通媒体管理器的重要功能之一。

用户应能够从单张图片、单个视频或指定画册中设置壁纸，也应能配置自动轮播规则。

### 8.2 功能需求

| 编号 | 需求名称 | 需求描述 | 优先级 |
|---|---|---|---|
| WR-01 | 设置单张壁纸 | 用户应能将选中图片设置为系统壁纸 | 高 |
| WR-02 | 设置视频壁纸 | 在支持平台上，用户应能设置视频壁纸 | 中 |
| WR-03 | 选择轮播来源 | 用户应能选择一个或多个画册作为轮播来源 | 高 |
| WR-04 | 随机轮播 | 系统应支持从轮播来源中随机选择壁纸 | 高 |
| WR-05 | 顺序轮播 | 系统应支持按画册顺序依次切换壁纸 | 中 |
| WR-06 | 轮播间隔 | 用户应能设置自动切换间隔 | 高 |
| WR-07 | 暂停 / 恢复 | 用户应能暂停或恢复壁纸轮播 | 高 |
| WR-08 | 壁纸历史 | 系统应记录最近设置过的壁纸 | 中 |
| WR-09 | 异常处理 | 若素材文件不存在或设置失败，应显示错误提示 | 高 |
| WR-10 | 平台适配 | 系统应根据 Windows、macOS、Linux、Android 的能力差异提供不同实现 | 高 |

### 8.3 壁纸模式需求

桌面端壁纸设置可能存在不同模式：

| 模式 | 说明 | 适用场景 |
|---|---|---|
| 原生模式 | 调用系统能力设置壁纸 | 稳定、资源占用低，适合普通图片壁纸 |
| 窗口模式 | 使用窗口覆盖或嵌入方式展示壁纸 | 适合部分动态壁纸或增强效果 |
| 移动端模式 | 适配 Android 壁纸和显示区域选择 | 适合移动设备 |

系统应在设置中清楚说明不同模式的差异，避免用户误解。例如窗口模式可能不是系统级壁纸，原生模式可能会覆盖用户原有系统壁纸。

### 8.4 壁纸轮播流程

1. 用户创建或选择一个画册。
2. 用户将图片或视频加入画册。
3. 用户进入壁纸设置。
4. 用户选择轮播来源、轮播方式和切换间隔。
5. 系统按规则切换壁纸。
6. 系统记录壁纸设置历史。
7. 若切换失败，系统显示错误原因。

## 九、数据需求

为了支撑插件扩展、画廊管理和壁纸轮播，系统需要维护以下数据：

| 数据类型 | 关键字段 | 用途 |
|---|---|---|
| 媒体文件 | ID、路径、文件名、大小、格式、媒体类型 | 画廊展示和文件管理 |
| 媒体属性 | 宽度、高度、宽高比、时长、缩略图路径 | 预览、筛选、排序 |
| 插件信息 | 插件 ID、名称、版本、描述、状态 | 插件管理 |
| 插件任务 | 任务 ID、插件 ID、配置、状态、日志、开始/结束时间 | 任务管理和错误排查 |
| 来源信息 | 来源 URL、插件 ID、任务 ID、下载时间 | 追踪素材来源 |
| 画册信息 | 画册 ID、名称、描述、封面、创建时间 | 画册管理 |
| 画册关系 | 画册 ID、媒体 ID、排序值 | 画册内容维护 |
| 壁纸配置 | 轮播来源、轮播方式、间隔、壁纸模式 | 壁纸轮播 |
| 壁纸历史 | 媒体 ID、设置时间、来源画册 | 历史记录和排序 |

## 十、非功能需求

| 类型 | 需求说明 |
|---|---|
| 易用性 | 普通用户应能在不理解插件脚本和数据库的情况下完成导入、浏览和设置壁纸 |
| 性能 | 大图库场景下应采用分页、缓存和索引，避免一次性加载全部素材 |
| 可靠性 | 媒体库、画册关系、插件配置和壁纸设置不应因异常退出轻易丢失 |
| 安全性 | 插件运行应有边界控制，异常插件不应导致主程序崩溃 |
| 可扩展性 | 新素材来源应优先通过插件扩展，而不是修改主程序 |
| 兼容性 | 不同平台可采用不同功能实现，但核心媒体库和画册数据模型应保持一致 |
| 可维护性 | 插件扩展、画廊管理、壁纸轮播、任务管理和平台适配应保持模块边界清晰 |
| 可诊断性 | 任务失败、网络失败、插件异常和壁纸设置失败应保留日志或错误信息 |

## 十一、约束与假设

### 11.1 约束

1. 项目应优先保持本地优先，不强制用户注册云端账号。
2. 插件获取第三方素材时，需要尊重站点规则和版权要求。
3. Windows、macOS、Linux 和 Android 的壁纸能力不同，不能保证所有平台功能完全一致。
4. 插件运行可能受网络环境、代理设置和目标站点反爬策略影响。
5. 大图库场景对性能要求较高，需要避免瀑布流或一次性加载过多素材导致卡顿。

### 11.2 假设

1. 用户拥有本地图片或视频素材，并希望长期管理。
2. 用户希望从多个来源收集壁纸或视觉素材。
3. 用户愿意安装插件来扩展新来源。
4. 用户更重视本地控制和可维护性，而不是强制云同步。
5. 高级用户可能需要 CLI 或批量处理能力。

## 十三、优先级总结

| 优先级 | 需求 |
|---|---|
| 第一优先级 | 本地导入、画廊浏览、画册管理、插件安装运行、图片壁纸设置、轮播来源选择 |
| 第二优先级 | 插件更新、视频壁纸、壁纸历史、复杂筛选、大图库性能优化 |
| 第三优先级 | Android 高级壁纸裁剪、插件权限细化、更多自动化 CLI 能力、跨设备同步 |

## 十四、总结

Kabegame 的核心需求可以概括为：通过插件扩展解决素材来源问题，通过画廊管理解决素材组织问题，通过壁纸轮播解决素材使用问题。

插件扩展决定了项目能否持续支持新的素材来源；画廊管理决定了用户能否长期维护大量媒体文件；壁纸轮播则让媒体库中的素材真正进入用户日常使用场景。三个模块相互依赖，构成 Kabegame 的核心产品闭环。
