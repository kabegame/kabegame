调研日期：2026 年 5 月  
姓名 / 学号：按课程要求补充

## 一、调研背景

随着图片、视频和壁纸素材数量不断增加，个人用户在素材收集、整理、检索和使用过程中会遇到明显的效率问题。传统做法通常依赖浏览器下载、文件夹分类、图片查看器预览和系统壁纸设置功能。这些工具各自独立，能够解决局部问题，但难以形成完整工作流。

Kabegame 的目标不是只做一个壁纸切换器，而是做一个可插件化、跨端、用户友好的视觉媒体管理 App。它需要同时解决两个核心需求：

1. **可插件化的跨端用户友好 App 需求**：用户希望在 Windows、macOS、Linux 和 Android 等平台上获得一致体验，并能通过插件扩展素材来源，而不是每次新增来源都修改主程序。
2. **媒体文件管理和元数据索引需求**：用户希望大量图片和视频能够被稳定导入、分类、搜索、筛选、预览、去重和复用，而不是散落在普通文件夹里。

因此，本调研报告围绕这两类需求展开，分析目标用户、典型场景、同类产品、技术路线和项目可行性。

## 二、调研目标

本次调研的目标如下：

1. 明确 Kabegame 的目标用户和核心使用场景。
2. 分析用户为什么需要插件化，而不是固定内置少量素材来源。
3. 分析跨端 App 在用户体验、平台能力和维护成本上的要求。
4. 分析媒体文件管理中对导入、分类、搜索、索引和备份的需求。
5. 调研同类产品的优点和不足，确定 Kabegame 的差异化定位。
6. 判断使用 Tauri、Rust、Vue、Rhai 插件和本地数据库等技术路线是否合理。

## 三、目标用户与使用场景

Kabegame 的目标用户主要包括以下几类：

| 用户类型 | 典型需求 | 主要痛点 |
|---|---|---|
| 壁纸收藏用户 | 收集、分类、预览和轮播大量壁纸 | 文件夹混乱、重复图片多、换壁纸流程繁琐 |
| 二次元 / 插画素材用户 | 从多个站点获取图片并长期整理 | 网站来源多、规则变化快、手动下载效率低 |
| 多端用户 | 在桌面和移动端浏览或使用素材 | 不同平台体验不一致，移动端操作不方便 |
| 高级用户 / 自动化用户 | 批量导入、脚本处理、命令行操作 | GUI 工具不够灵活，独立脚本又难维护 |
| 插件开发者 | 为新网站或新导入方式编写扩展 | 缺少统一插件格式、配置和运行环境 |

典型使用场景如下：

1. 用户从本地文件夹导入大量图片和视频，应用自动生成缩略图并加入媒体库。
2. 用户安装某个素材来源插件，通过配置页输入页数、关键词或分类后开始收集。
3. 下载任务运行时，用户能够看到进度、日志、失败原因和已收集结果。
4. 用户将喜欢的图片加入画册，并将画册设置为壁纸轮播来源。
5. 用户按照插件来源、日期、尺寸、宽高比、媒体类型、收藏状态等条件筛选素材。
6. 用户在 Android 端浏览素材，或对图片进行移动端壁纸显示区域选择。
7. 高级用户通过 CLI 执行批量导入、插件运行或自动化处理。

## 四、需求一：可插件化的跨端用户友好 App

### 4.1 插件化需求

视觉素材来源具有明显的不稳定性：不同网站的页面结构、接口规则、反爬策略、权限要求和网络访问条件都可能变化。如果把所有来源逻辑都写死在主程序中，主程序会频繁修改、测试和发布，维护成本较高。

因此，Kabegame 需要插件化架构。插件化的核心需求包括：

| 需求 | 说明 |
|---|---|
| 插件安装 | 用户可以从插件商店、插件索引或本地 `.kgpg` 文件安装插件 |
| 插件配置 | 插件可以声明参数，例如页数、关键词、质量、分类、登录态或请求头 |
| 插件运行 | 插件能够执行素材获取、文件导入或批量处理任务 |
| 插件更新 | 插件来源规则变化后，可以单独更新插件，而不必更新主程序 |
| 插件禁用 / 删除 | 用户可以停用不可信或失效插件 |
| 插件文档 | 插件应提供使用说明、网络要求和参数解释 |
| 插件安全 | 插件访问网络和本地文件时需要边界控制和用户提示 |
| 插件版本兼容 | 插件需要声明版本和兼容性，避免旧插件在新主程序中异常运行 |

插件化带来的价值是：主程序保持稳定，内容来源通过插件扩展。这样既能支持更多站点和导入方式，也能降低核心应用的维护压力。

### 4.2 跨端需求

Kabegame 面向桌面端和移动端。Tauri 官方说明其可以使用 Web 前端和 Rust 后端构建跨平台应用，覆盖 Linux、macOS、Windows、Android 和 iOS 等平台。这说明使用一套核心技术支持多端应用具有可行性。

跨端需求包括：

| 平台 | 需求重点 |
|---|---|
| Windows | 安装包、壁纸设置、快捷键、代理检测、多显示器、虚拟磁盘 |
| macOS | 应用权限、文件夹访问、Gatekeeper、壁纸设置、虚拟文件夹 |
| Linux | 发行版依赖、Wayland / X11、KDE 壁纸插件、FUSE |
| Android | 移动端浏览、壁纸预览、图片区域裁剪、后台权限 |
| CLI | 批处理、自动化、插件运行和外部工具集成 |

跨端并不意味着所有平台功能完全一致。桌面端可以支持虚拟磁盘、CLI 和系统壁纸深度集成；移动端则应优先保证浏览、预览和基础壁纸操作。跨端设计的重点是保持核心数据模型一致，而不是强行让所有平台拥有相同界面。

### 4.3 用户友好需求

插件化和跨端能力会增加复杂度，因此用户友好性非常重要。Kabegame 需要把复杂能力包装成清晰的用户流程。

用户友好需求包括：

1. 首次启动时提供清晰的导入、插件安装和壁纸设置入口。
2. 插件运行失败时显示可理解的错误原因，例如网络错误、403、代理未生效、插件过期等。
3. 设置项使用用户能理解的语言，不让普通用户直接面对过多技术细节。
4. 对需要系统权限的功能进行明确提示，例如文件夹访问、虚拟磁盘、壁纸设置和后台任务。
5. 保持桌面端和移动端信息结构一致，例如画廊、画册、任务、插件、设置。
6. 提供帮助文档和插件说明，减少用户在 GitHub issue 中反复询问同类问题。
7. 对高级功能采用渐进式展示，普通用户先看到核心流程，高级用户再进入配置和自动化功能。

## 五、需求二：媒体文件管理与元数据索引

### 5.1 媒体文件管理需求

媒体管理是 Kabegame 的核心基础。如果没有稳定的本地媒体库，插件获取、画册整理、壁纸轮播和虚拟磁盘都会缺少统一数据基础。

媒体文件管理需求包括：

| 需求 | 说明 |
|---|---|
| 本地导入 | 支持图片、视频、文件夹、压缩包等导入方式 |
| 媒体库保存 | 将导入或获取的素材纳入统一本地库 |
| 缩略图生成 | 为图片和视频生成预览，提升浏览效率 |
| 画册管理 | 支持创建画册、加入图片、排序、封面和描述 |
| 收藏与标记 | 支持收藏、评分、标签或其他用户标记 |
| 去重识别 | 通过文件哈希、尺寸或元数据辅助识别重复文件 |
| 批量操作 | 支持批量导入、删除、加入画册、修改信息 |
| 虚拟目录 | 将画册或筛选结果映射为虚拟文件夹 / 虚拟磁盘 |
| 备份友好 | 媒体文件和数据库结构应便于备份和迁移 |

媒体管理不应只依赖物理文件夹，因为同一张图片可能同时属于多个画册、多个标签或多个筛选结果。应用需要在文件系统之外建立结构化索引。

### 5.2 元数据索引需求

同类产品调研显示，成熟媒体管理工具通常非常重视元数据。digiKam 使用数据库保存标签和评论等元数据，并支持按时间、文件夹和集合组织照片；Adobe Bridge 也强调标签、评分、元数据、关键词、过滤和搜索；Immich 则使用数据库支持元数据搜索和上下文搜索。

Kabegame 的元数据索引需求包括：

| 元数据类型 | 示例 | 用途 |
|---|---|---|
| 基础文件信息 | 文件名、路径、大小、格式、创建时间、修改时间 | 浏览、排序、查找、去重 |
| 媒体属性 | 宽度、高度、宽高比、媒体类型、时长 | 图片 / 视频区分、横竖屏筛选 |
| 来源信息 | 插件 ID、来源 URL、任务 ID、下载时间 | 追踪素材来源、排查插件问题 |
| 用户信息 | 画册、收藏、标签、评分、备注 | 个性化整理和复用 |
| 技术元数据 | EXIF、XMP、IPTC、GPS、相机信息 | 高级搜索、摄影信息查看 |
| 任务元数据 | 任务状态、日志、失败原因、运行参数 | 任务复盘、错误诊断 |
| 壁纸历史 | 设置时间、轮播来源、使用次数 | 壁纸轮播和历史追踪 |

元数据索引应支持以下查询能力：

1. 按画册查询。
2. 按插件来源查询。
3. 按日期、年月日查询。
4. 按图片 / 视频类型查询。
5. 按尺寸、宽高比、文件格式查询。
6. 按收藏、壁纸历史或任务结果查询。
7. 按关键词、文件名或描述查询。
8. 组合筛选和排序，例如“某插件来源 + 横屏 + 最近下载 + 视频排除”。

### 5.3 索引性能需求

媒体库规模变大后，性能会成为关键问题。用户可能拥有几千到几万张图片，如果每次浏览都直接扫描文件夹，会导致加载慢、排序慢、搜索慢。

因此，Kabegame 需要本地数据库和索引机制：

1. 图片导入时写入数据库，而不是每次启动重新扫描。
2. 缩略图和媒体信息应缓存，避免重复解析。
3. 画廊浏览应使用分页或虚拟列表，避免一次性渲染大量图片。
4. 常用筛选字段应建立索引，例如插件来源、画册、日期、媒体类型、宽高比。
5. 数据库迁移需要安全，避免版本升级导致媒体库损坏。
6. 元数据和文件路径应保持可恢复，方便用户备份和迁移。

## 六、同类产品调研

| 产品 / 类型 | 优点 | 不足 | 对 Kabegame 的启示 |
|---|---|---|---|
| 系统自带壁纸功能 | 稳定、简单、无需学习 | 只负责设置壁纸，不管理素材来源和元数据 | Kabegame 需要整合素材库和壁纸使用流程 |
| 普通文件管理器 | 操作直观，符合用户习惯 | 缺少跨画册分类、插件来源、任务历史和高级索引 | 不能只依赖文件夹，需要结构化媒体库 |
| digiKam | 专业照片管理，支持标签、数据库和丰富元数据 | 更偏摄影工作流，不聚焦插件化素材获取和壁纸轮播 | 证明本地数据库和元数据管理是必要能力 |
| Adobe Bridge | 创意资产管理能力强，支持元数据、关键词、标签、评分和高级搜索 | 偏 Adobe 生态和专业创意工作流 | 说明资产管理应重视预览、批量操作和元数据过滤 |
| Immich | 支持照片 / 视频管理、移动端备份、元数据和智能搜索 | 更偏自托管服务，不是本地桌面壁纸工作流 | 说明跨端浏览、搜索和移动端体验很重要 |
| 独立爬虫脚本 | 灵活，适合高级用户 | 无统一界面、配置和任务管理，难以维护 | 插件化应保留脚本灵活性，同时提供用户友好界面 |
| Wallpaper Engine 等壁纸软件 | 壁纸展示能力强，动态壁纸生态成熟 | 更重展示和创作，不重视本地素材索引与插件采集 | Kabegame 应突出“素材管理 + 壁纸应用”的结合 |

同类产品调研表明，市场上已有许多局部解决方案，但它们通常只解决其中一部分问题。Kabegame 的差异化定位应是：**本地优先、插件可扩展、跨端可用、围绕视觉素材管理和壁纸应用的一体化工具**。

## 七、技术方案调研

| 技术 / 方案 | 用途 | 适配原因 |
|---|---|---|
| Tauri | 跨平台桌面 / 移动应用框架 | 支持 Web 前端和 Rust 后端，适合跨平台 App |
| Vue 3 | 图形界面 | 适合构建画廊、任务、插件、设置等复杂交互 |
| Rust | 后端核心 | 适合处理文件系统、数据库、下载任务、CLI 和平台能力 |
| Rhai | 插件脚本 | 易嵌入 Rust，适合作为插件运行语言 |
| 本地数据库 | 媒体库与元数据索引 | 支持高效查询、筛选、排序和数据迁移 |
| CLI | 自动化能力 | 支持高级用户批处理和外部脚本调用 |
| 虚拟磁盘 / 虚拟文件夹 | 文件系统访问 | 让用户按画册或筛选结果访问素材 |

该技术组合能够较好地覆盖项目需求：前端负责用户友好界面，Rust 负责系统能力和数据可靠性，Rhai 插件负责可扩展素材来源，本地数据库负责媒体文件和元数据索引。

## 八、可行性分析

### 8.1 技术可行性

Tauri、Rust 和 Vue 的组合可以实现跨平台桌面应用，项目现有结构也已经包含桌面端、Android 预览版、CLI、插件系统和本地媒体库。因此，从技术上看，可插件化跨端 App 是可行的。

媒体文件管理和元数据索引也具有可行性。成熟媒体管理工具已经证明，数据库、标签、元数据和高级搜索是管理大量媒体文件的有效方式。Kabegame 可以采用更轻量的本地优先方案，围绕壁纸和视觉素材场景做针对性设计。

### 8.2 用户需求可行性

用户反馈和项目 issue 表明，用户确实关心 Windows 壁纸体验、插件源可用性、代理问题、画廊分页、Android 显示和文档一致性。这些反馈说明 Kabegame 的核心方向是有真实需求的。

### 8.3 维护可行性

插件化可以降低主程序维护压力，但也会引入插件版本、插件安全和文档维护问题。因此，项目需要在插件自由度和安全控制之间保持平衡。

可行的维护策略包括：

1. 主程序只维护稳定插件运行时和媒体库核心。
2. 插件通过独立仓库、独立版本和独立文档维护。
3. 对插件建立兼容性标记和错误诊断机制。
4. 对常见网络问题提供统一帮助和代理设置。

## 九、风险与约束

| 风险 | 说明 | 应对措施 |
|---|---|---|
| 跨平台差异 | Windows、macOS、Linux、Android 的权限和系统 API 不一致 | 建立平台测试矩阵，功能分层支持 |
| 插件安全 | 插件可能访问网络和本地文件 | 增加权限提示、插件校验、禁用和删除机制 |
| 插件失效 | 第三方网站规则变化可能导致插件不可用 | 插件独立更新，提供错误日志和版本管理 |
| 网络代理复杂 | Pixiv 等来源可能需要代理，系统代理检测不稳定 | 提供代理检测、手动代理设置和文档说明 |
| 大图库性能 | 大量图片会导致加载、搜索和缩略图生成变慢 | 使用数据库索引、分页、缓存和后台任务 |
| 数据迁移风险 | 数据库结构升级可能影响用户媒体库 | 增加迁移测试、备份机制和回滚说明 |
| 文档过期 | 应用迭代快，帮助文档可能落后 | 发布前检查文档，建立贡献指南 |
| 版权与合规 | 用户获取的素材可能涉及版权和站点条款 | 明确用户责任，避免内置违规行为 |

## 十、调研结论

通过调研可以得出结论：Kabegame 的核心需求应围绕“可插件化跨端用户友好 App”和“媒体文件管理与元数据索引”展开。

第一，插件化是必要的。视觉素材来源变化快，不适合把所有获取逻辑写死在主程序中。插件系统可以让 Kabegame 支持更多来源，同时保持核心应用稳定。

第二，跨端体验是重要竞争点。桌面端负责深度文件管理、壁纸设置、虚拟磁盘和 CLI；移动端负责浏览、预览和轻量壁纸操作。不同平台功能可以分层，但核心媒体库模型应保持一致。

第三，媒体管理和元数据索引是项目的基础能力。没有稳定的本地媒体库、缩略图缓存、画册结构和元数据索引，插件获取和壁纸轮播都会变成分散功能，难以形成完整产品。

第四，用户友好性决定项目能否被普通用户接受。插件、代理、权限、任务失败和跨平台差异都比较复杂，必须通过清晰界面、错误提示、帮助文档和诊断工具降低使用门槛。

因此，Kabegame 的合理定位是：**一个本地优先、插件可扩展、跨端可用、以媒体文件管理和元数据索引为核心的视觉素材与壁纸管理工具**。
